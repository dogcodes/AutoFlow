# AutoFlow 激活码格式与验证规则

## 概览

AutoFlow 当前使用长度为 **24 位** 的本地激活码，仅靠本地校验即可判断有效性。格式为：

```
DDD YYMMDD VV T SSSS CCCCCCCC
```

每段含义：

| 区段 | 长度 | 内容 | 说明 |
| --- | --- | --- | --- |
| `DDD` | 3 | 授权天数 | 1~999，代表激活完成后可使用的总天数 |
| `YYMMDD` | 6 | 生成日期 | 格式为两位年、两位月、两位日（非 8 位），用来算码是否已过期 |
| `VV` | 2 | 激活码有效期 | 表示激活码自生成日起的可用天数（自然日，1 表示当天内有效，3 表示 3 天内） |
| `T` | 1 | 类型位 | 0=共用（不限 App）、1=AutoFlow App、2=抢红包 App、3=测试/内部，便于在前端提示或后台统计 |
| `SSSS` | 4 | 随机种子 | 由用户输入的 seed（只保留字母/数字）截取前 4 位，不足补 `0` |
| `CCCCCCCC` | 8 | 校验码 | MD5（前 16 位 + 设备 ID），取前 8 个字节（16 个十六进制字符） |

## 生成规则

- 天数 `DDD`、有效期 `VV` 保持在 [1, 999] / [1, 30] 范围；类型 `T` 强制在 [0, 3] 内。
- 生成时会用 `LocalDate.now()` 生成 `YYMMDD`，种子只保留字母+数字并截取 4 位。
- 校验码 = `MD5(data + deviceId)`，其中 `data` 是前 16 位 `DDDYYMMDDVVTSSSS`。
- 最终输出 `data + checksum`，长度固定为 24 位。

## 校验流程

1. 检查长度为 24、设备 ID 非空。
2. 拆分 `DDD/YYMMDD/VV/T/SSSS` 与 `CCCCCCCC`。
3. `YYMMDD` 需能通过 `DateTimeFormatter.ofPattern("yyMMdd")` 解析；`VV` 在 1~30，`T` 在 0~3。
4. 计算截止日 = 生成日 + `VV - 1` 天，按自然日比较；超期则视为无效。
5. 当前 AutoFlow App 只接受类型 `T` 为 0（共用）或 1（AutoFlow 专用），其他值（例如 2=抢红包 App）会视为不匹配。
6. 用前 16 字符（`DDDYYMMDDVVTSSSS`）+设备 ID 生成 MD5，取前 8 字节，与 `CCCCCCCC` 比对。

## 激活失败原因

激活失败会根据原因返回不同提示（见 `LicenseDialog` 中 `FailureReason.toMessage()`）：

- `FORMAT_ERROR`：激活码长度/字段不对，提示“激活码格式错误”。
- `EXPIRED`：已超过有效期，提示“激活码已过期”。
- `TYPE_MISMATCH`：类型位不是 0/1，提示“此激活码不适用于当前应用”。
- `ALREADY_USED`：该激活码已经记录在 `activated_keys`，提示“激活码已被使用”。
- `DEVICE_MISMATCH`：校验码和设备 ID 不一致，提示“设备与激活码不匹配”。
- `UNKNOWN`：其它不可识别的问题，提示“激活码无效”。
5. 用前 16 字符（`DDDYYMMDDVVTSSSS`）+设备 ID 生成 MD5，取前 8 字节，与 `CCCCCCCC` 比对。

## 旧码兼容

旧版 20 位码已弃用（`SSSSS` + 4 位校验）；新版仅尝试 24 位逻辑。若必须兼容，可在 `LicenseManager` 先尝试新规则，失败再尝试旧规则并迁移 `SharedPreferences`。

## 调试指南

- 在“许可证管理”对话框长按设备 ID 可打开 `LicenseDebugActivity`（仅 debug 构建）。
- 可设置设备 ID、使用天数、有效期、类型与随机种子，点击生成即可获得 24 位激活码。
- 校验区也会要求 24 位并验证类型位是否在合法范围。

## 建议

- 记录每个激活码的类型位，便于区分体验码 / 正式码 / 活动码 / 测试码的来源。
- 调试页默认 type=1（正式码），必要时可快速切换。
- 若未来需要更多类型可用后端配置或改用多位类型字段。
